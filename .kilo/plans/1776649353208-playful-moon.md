# Enhance Step 2: Matrix

## Goal
Improve `Step 2: Matrix` so entering decision values is faster, clearer, and easier to verify, while keeping the existing TOPSIS computation flow unchanged.

## Current state
- `src/components/topsis/Step2Matrix.tsx` renders a simple table with either numeric `Input` fields or linguistic `Select` fields.
- Validation is performed centrally in `src/pages/Index.tsx` and only blocks on Next; users do not get much guidance before submission.
- A separate `Numeric Preview` table exists, but it is passive and does not help users detect incomplete rows, suspicious values, or normalization blockers.
- Step 1 already has richer UX patterns (bulk actions, summary cards, clearer helper text), so Step 2 currently feels behind the rest of the flow.

## Scope chosen
Prioritize these three areas:
1. Faster data entry
2. Validation clarity
3. Analysis preview

Do not change:
- TOPSIS formulas in `src/lib/topsis.ts`
- Step order or overall wizard structure in `src/pages/Index.tsx`
- Weight logic in Step 3

## Implementation plan

### 1. Refactor Step 2 into clearer sections
- Split `Step2Matrix` into three visual zones:
  - Header / helper actions
  - Editable matrix
  - Preview / health summary
- Keep the component in one file initially unless it grows past reasonable size; if it does, extract focused subcomponents such as `MatrixGrid`, `MatrixToolbar`, and `MatrixPreviewCard`.

### 2. Add spreadsheet-style data-entry helpers
- Add row and column helper actions where safe and obvious:
  - clear row
  - clear column
  - fill column with a repeated value
  - optionally copy first row / previous row pattern
- Add bulk paste support for numeric criteria using tab/newline parsing from spreadsheets.
- Support keyboard flow for numeric cells:
  - Enter or Arrow keys move focus
  - Tab remains natural
- For linguistic criteria, keep `Select`, but make the trigger width and labels more compact so larger matrices remain readable.

### 3. Improve validation feedback before Next
- Surface completion state live inside Step 2 instead of only after clicking Next.
- Show a compact matrix status summary:
  - filled cells vs total cells
  - invalid cells count
  - empty cells count
- Highlight incomplete rows and columns visually.
- Pre-check zero-only columns and warn early because `checkZeroColumns` currently blocks later in `computeAllThroughRanking`.
- Preserve the existing authoritative validation in `src/pages/Index.tsx`; Step 2 should add guidance, not replace final guards.

### 4. Upgrade the numeric preview into a useful analysis panel
- Keep the raw numeric preview, but add lightweight diagnostics per criterion:
  - min
  - max
  - count filled
  - criterion type badge (`benefit` / `cost`)
- For linguistic criteria, make the label-to-value mapping visible near the header so users understand what each selection becomes numerically.
- Add a small “ready to continue” state when all cells are valid and no zero-column issue exists.

### 5. Improve large-matrix usability
- Add sticky first column for alternative names on wider screens.
- Add sticky header row if supported by existing table styling.
- Keep horizontal scrolling, but ensure action controls remain visible above the table.
- For mobile, avoid forcing the full matrix into tiny cells; preferred fallback is stacked criterion summaries or a horizontally scrollable table with clearer spacing.

### 6. Align state management with the richer UI
- Extend `Step2Matrix` props only as needed; prefer derived values from existing inputs.
- Likely compute inside `Index.tsx` and pass down:
  - completion counts
  - zero-column warning info
  - optional per-column stats from `rawMatrix`
- Keep `matrixInputs` as the source of truth so current numeric/linguistic conversion behavior remains intact.
- If focus management becomes complex, use refs local to `Step2Matrix` rather than lifting transient UI state to `Index.tsx`.

### 7. Verify behavior with targeted tests
- Add component tests for:
  - rendering numeric and linguistic cells correctly
  - inline status changes when values are edited
  - bulk paste parser for tabular numeric data
  - zero-column warning visibility
- Add at least one integration-style test across `Index.tsx` to confirm Step 2 still blocks invalid progression and still passes valid data into later steps.

## Suggested delivery order
1. Add live status + validation summary
2. Improve preview panel with criterion diagnostics
3. Add bulk paste and simple clear/fill helpers
4. Add keyboard navigation and focus handling
5. Polish responsiveness and sticky table behavior
6. Add/update tests

## Files likely to change
- `src/components/topsis/Step2Matrix.tsx`
- `src/pages/Index.tsx`
- `src/lib/topsis.ts` only if small read-only helper utilities are worth extracting for stats; avoid changing math behavior
- `src/test/*` for new Step 2 coverage

## Risks and guardrails
- Bulk paste can conflict with linguistic columns; safest default is numeric-only paste with clear messaging for mixed matrices.
- Keyboard navigation can become brittle with `Select` components; keep navigation enhancements limited to numeric inputs first.
- Sticky headers/columns may require careful Tailwind class tuning to avoid breaking overflow behavior.
- Avoid duplicating validation logic in too many places; shared derived helpers are better than ad hoc checks.

## Acceptance criteria
- Users can complete medium-size matrices materially faster than with the current per-cell-only workflow.
- Users can see incomplete/invalid areas before clicking Next.
- Users can understand the numeric meaning of linguistic selections immediately.
- Users get early warning when a criterion column will fail normalization because it is all zeros.
- Existing TOPSIS results remain unchanged for the same valid inputs.

## Recommended default implementation stance
- Start with low-risk UX improvements inside existing architecture.
- Keep `Step2Matrix` backward-compatible with current props where possible.
- Treat bulk paste and live summaries as the biggest value-for-effort improvements.
